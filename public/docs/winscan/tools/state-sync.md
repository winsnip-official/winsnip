# State Sync Configuration

Generate state sync configuration to quickly sync your Cosmos node.

## Overview

State sync allows you to sync a Cosmos node in minutes instead of days by downloading a snapshot of the blockchain state at a specific height. WinScan automatically generates the configuration you need.

## What is State Sync?

State sync is a feature that allows new nodes to:
- Skip downloading the entire blockchain history
- Start from a recent snapshot
- Sync in minutes instead of hours/days
- Save disk space and bandwidth

## How to Use

### 1. Access State Sync Tool

Navigate to any chain and click "Network" → "State Sync" in the sidebar, or visit:
```
https://winscan.org/[chain]/statesync
```

### 2. Configure Your Node

Enter your node information:
- **Service Name**: Your systemd service name (e.g., `kiichaind`)
- **Home Directory**: Your node's home directory (e.g., `~/.kiichain`)

### 3. Copy Configuration

WinScan generates the complete configuration:

```bash
# Stop your node
sudo systemctl stop kiichaind

# Reset node data (keeps priv_validator_state.json)
kiichaind tendermint unsafe-reset-all --home ~/.kiichain --keep-addr-book

# Configure state sync in config.toml
[statesync]
enable = true
rpc_servers = "https://rpc1.example.com:26657,https://rpc2.example.com:26657"
trust_height = 1234567
trust_hash = "ABC123..."
trust_period = "168h0m0s"

# Add persistent peers
persistent_peers = "node1@ip:26656,node2@ip:26656"

# Start your node
sudo systemctl start kiichaind
```

### 4. Apply Configuration

1. Copy the generated commands
2. Run them on your server
3. Wait for sync to complete (usually 5-15 minutes)

## Configuration Parameters

### Trust Height

The block height of the snapshot:
- Should be recent (within last 24 hours)
- Must be a snapshot block
- Automatically selected by WinScan

### Trust Hash

The block hash at trust height:
- Verifies snapshot integrity
- Automatically fetched from RPC
- Must match the actual block hash

### RPC Servers

List of RPC endpoints for state sync:
- Minimum 2 servers recommended
- Should be reliable and fast
- Automatically selected from available endpoints

### Persistent Peers

Active nodes to connect to:
- Helps with P2P connectivity
- Automatically fetched from network
- Filtered for active and listening peers

## Step-by-Step Guide

### Prerequisites

- A server with sufficient resources
- Node binary installed
- Firewall configured (port 26656 open)

### 1. Install Node Binary

```bash
# Example for Kiichain
git clone https://github.com/KiiChain/kiichainV3
cd kiichainV3
make install
```

### 2. Initialize Node

```bash
kiichaind init "your-moniker" --chain-id kiichainV3-mainnet
```

### 3. Download Genesis

```bash
wget -O ~/.kiichain/config/genesis.json https://example.com/genesis.json
```

### 4. Apply State Sync Config

Use the configuration generated by WinScan:

```bash
# Stop node if running
sudo systemctl stop kiichaind

# Reset data
kiichaind tendermint unsafe-reset-all --home ~/.kiichain --keep-addr-book

# Edit config.toml with generated values
nano ~/.kiichain/config/config.toml
```

### 5. Start Node

```bash
sudo systemctl start kiichaind
sudo journalctl -u kiichaind -f
```

## Monitoring Sync Progress

Check if state sync is working:

```bash
# Check logs
sudo journalctl -u kiichaind -f | grep -i "state sync"

# Check sync status
kiichaind status 2>&1 | jq .SyncInfo
```

You should see:
- "Discovering snapshots"
- "Applying snapshot chunk"
- "State sync complete"

## Troubleshooting

### State Sync Fails

**Problem**: Node can't find snapshots

**Solution**:
- Try different RPC servers
- Increase `trust_period`
- Check if RPC servers support state sync

### Snapshot Verification Failed

**Problem**: Trust hash doesn't match

**Solution**:
- Regenerate configuration with new trust height
- Verify RPC servers are synced
- Check network connectivity

### No Peers Found

**Problem**: Node can't connect to peers

**Solution**:
- Add more persistent peers
- Check firewall settings (port 26656)
- Verify seeds are correct

### Sync Stuck at Specific Height

**Problem**: Node stops syncing

**Solution**:
- Restart the node
- Try regular sync instead
- Check disk space

## Advanced Configuration

### Custom Trust Period

```toml
[statesync]
trust_period = "336h0m0s"  # 14 days
```

### Snapshot Discovery

```toml
[statesync]
discovery_time = "15s"
chunk_request_timeout = "10s"
chunk_fetchers = "4"
```

### Temp Directory

```toml
[statesync]
temp_dir = "/tmp/statesync"
```

## Best Practices

1. **Use Recent Snapshots**: Trust height should be within last 24 hours
2. **Multiple RPC Servers**: Use at least 2 reliable RPCs
3. **Backup priv_validator_state.json**: Never lose this file
4. **Monitor Logs**: Watch for errors during sync
5. **Verify Sync**: Check block height after completion

## Comparison: State Sync vs Regular Sync

| Feature | State Sync | Regular Sync |
|---------|-----------|--------------|
| Time | 5-15 minutes | Hours to days |
| Disk Space | Minimal | Full history |
| Bandwidth | Low | High |
| Historical Data | No | Yes |
| Complexity | Medium | Low |

## When to Use State Sync

✅ **Use State Sync When:**
- Setting up a new validator
- Recovering from corruption
- Testing on testnet
- Don't need historical data

❌ **Don't Use State Sync When:**
- Need full blockchain history
- Running an archive node
- Providing historical queries
- Debugging old transactions
